# Feature: 1.1.1 Create Elixir Project Structure

## Problem Statement

The jido_code project needs a properly structured Elixir application with supervision tree support and all required dependencies configured. This is the foundation for the entire agentic coding assistant TUI.

### Impact Analysis
- **Scope**: Creates the base application structure that all future phases build upon
- **Dependencies**: None (first task)
- **Risk**: Low - standard Elixir project setup

## Solution Overview

Generate a supervised Elixir application and configure all dependencies specified in the proof-of-concept plan:

### Core Dependencies
- `jido` (~> 1.2) - Autonomous agent framework (from Hex)
- `jido_ai` (~> 0.5.3) - AI/LLM integration (local path)
- `term_ui` - TUI framework (local path)
- `phoenix_pubsub` (~> 2.1) - Agent-TUI communication
- `jason` - JSON encoding/decoding
- `luerl` - Lua sandbox for tool execution
- `rdf` (~> 2.0) - Knowledge graph (RDF)
- `libgraph` (~> 0.16) - In-memory graph algorithms

### Development Dependencies
- `ex_doc` - Documentation generation
- `credo` - Static analysis
- `dialyxir` - Type checking

## Technical Details

### File Locations
- `mix.exs` - Project configuration and dependencies
- `lib/jido_code.ex` - Main module
- `lib/jido_code/application.ex` - OTP Application with supervision tree
- `config/config.exs` - Base configuration
- `config/dev.exs` - Development configuration
- `config/test.exs` - Test configuration
- `config/prod.exs` - Production configuration
- `config/runtime.exs` - Runtime/environment-specific configuration

### Key Decisions
1. Created project structure directly (not using `mix new` since directory existed)
2. Reference `term_ui` via local path (`../term_ui`)
3. Reference `jido_ai` via local path (`../agentjido/jido_ai`)
4. Use `jido` from Hex (~> 1.2) instead of local path
5. Configure Phoenix PubSub and Registry in application supervisor
